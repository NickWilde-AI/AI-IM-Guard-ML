# 深挖问答：IM 私聊违规审核模型

这份文档用于准备面试官连续追问。回答原则：先给结论，再讲原因，最后补工程兜底或实验依据。

## 1. 你这个项目到底解决了什么问题？

解决的是 IM 私聊风控里“只看文本不够、只判二分类不够、不能直接接处置系统”的问题。原系统能判断有没有违规，但不能稳定判断风险严重程度，也不能融合打赏、进房、关注、登录等行为信号。我的方案把输入升级成语义证据 + 行为证据，把输出升级成风险等级、违规判定、处置建议和解释依据。

## 2. 为什么不是普通内容安全分类？

普通内容安全分类通常只看 prompt 或 response 文本，输出 safe/unsafe。这个项目面向直播 IM 私聊，有业务行为证据和处置链路，输出要能直接被策略系统、人审系统和数据看板消费。所以它更像业务审核 Judge，而不是通用 moderation classifier。

## 3. 为什么行为证据这么关键？

IM 灰产很多时候会规避关键词。比如“今晚老规矩”文本本身很正常，但如果同一时间出现陌生用户短时大额打赏，就可能是代刷包榜。纯文本模型抓不到这种模式，规则引擎单看行为阈值也容易误杀，必须联合判断。

## 4. 行为特征为什么不直接喂原始日志？

原始日志太长、太噪，直接进 prompt 会超上下文，也会把模型注意力浪费在无关事件上。生产里应该由行为特征服务先聚合成摘要，例如近 7 日打赏金额、短时间进房次数、是否异地登录、是否批量投递，再让模型做证据融合。

## 5. 为什么输出要拆成三个标签？

`final_judgment` 解决有没有违规，`risk_level` 解决严重程度，`handling_suggestion` 解决怎么处置。三者消费方不同，不能互相替代。尤其是 `risk_level` 到 `handling_suggestion` 不是线性映射，高风险但行为证据不足时不一定直接封号。

## 6. 为什么不用一个二分类模型加规则映射？

二分类只能学“有没有违规”，学不到业务处置边界。规则映射会把很多灰区粗暴线性化，比如 high 就 ban，但真实业务里 ban 要看证据链完整性和误杀成本。多任务 SFT 能让模型直接学习处置档位。

## 7. 为什么不用三个模型分别预测？

三个模型会带来三类问题：推理成本高、延迟高、输出不一致。比如一个模型判不违规，另一个模型给 ban，后处理会很复杂。单模型多任务能共享语义和行为理解表示，也能让输出协议天然一致。

## 8. 为什么选 Qwen3.5-27B？

不是盲目选大模型，而是看灰区能力、部署成本和稳定性。8B 在 mid_risk 和 handling 上掉点明显；MoE 成本更优但效果略低；27B 稠密版在风险分级、处置建议和部署复杂度之间比较平衡。

## 9. 为什么 completion-only loss？

prompt 里有大量 rubric、处置策略表和 JSON Schema。如果对 prompt 也算 loss，模型会浪费容量学习复述输入。completion-only 只对 assistant 输出 JSON 算 loss，监督信号更集中。

## 10. 公开数据没有处置标签，怎么训练？

公开数据只参与 `final_judgment` 二分类辅助任务，不参与 `risk_level` 和 `handling_suggestion` 的 loss。这样它补强通用文本安全能力，但不会污染业务处置档位。

## 11. 合成数据会不会把模型带偏？

会有风险，所以要控制。第一，生成器 seed 来自真实工单；第二，生成结果走 JSON Schema 和证据一致性校验；第三，合成数据只作为补长尾和边界样本，不替代真实工单；第四，最终用纯人工测试集验证。

## 12. 灰区 refinement 为什么有效？

因为它不是随机加数据，而是针对当前模型最容易错的区域加数据。比如模型把“语义淡但行为强”的违规样本判成安全，这类样本回灌后能直接补模型短板。相比单纯堆量，hard sample 更有效。

## 13. committee 为什么要有规则引擎？

因为需要异构判断源。self 是当前模型，Qwen-flash 是另一个 LLM，规则引擎是非 LLM。三个来源互相独立，能降低把错误金标或噪声样本回灌进训练集的概率。

## 14. 怎么防止测试集泄漏？

三层防护：ticket_id 集合差保证工单不重复；合成样本用独立前缀；训练前做近重复检测，例如 SimHash 或 embedding 相似度，超过阈值就剔除。面试里要强调测试集是模型选择和上线判断的底线。

## 15. 为什么看 macro-F1？

类别不均衡，`ignore` 样本最多，`ban_account` 样本少但业务代价最高。micro-F1 容易被大类掩盖，macro-F1 能更真实反映少数关键类别的能力。

## 16. 为什么还看 ban_account FPR？

因为封号误杀直接产生客诉，是业务最敏感指标。一个模型即使 overall accuracy 高，只要 ban FPR 高，也不能上线。这个指标要单独设红线。

## 17. 为什么 AUPRC 比 ROC-AUC 更适合？

线上违规样本通常占比低，ROC-AUC 在类别不均衡时容易虚高。AUPRC 直接看 precision-recall，更贴近业务调阈值时的取舍。

## 18. JSON 输出坏了怎么办？

先截取完整 JSON object 解析；失败后用正则提取枚举字段；仍失败就默认安全档并打日志。重处罚不能依赖坏格式输出，宁可降级到规则或人审。

## 19. 模型给 ban 会不会直接封号？

不会。模型只给 `handling_suggestion=ban_account`，生产动作是进入人审复核。最终封号由人审和策略系统决定。这样能利用模型召回高风险，又控制误杀成本。

## 20. 线上最大风险是什么？

最大风险是上游行为特征污染。例如礼物金额默认值写错，会导致模型误判高风险。所以线上必须监控特征分布，关键字段异常时要拦截重处罚或切回兜底。

## 21. 如果线上效果和离线不一致，怎么排查？

按链路排查：输入分布是否变了，聊天证据条数是否变短，行为字段是否缺失，prompt/rubric 版本是否一致，JSON 解析是否失败，策略路由是否改过。不要上来就认为是模型权重问题。

## 22. 为什么 shadow 模式必要？

shadow 阶段模型只预测不执行，用来观察线上真实分布、解析失败率、和规则/人审差异。很多线上问题，比如证据截断不一致、字段缺失、行为特征异常，离线测试发现不了。

## 23. 如果业务新增违规主题怎么办？

先补 taxonomy 和 rubric，再抽历史样本或构造合成样本，更新 schema 和 prompt，训练新版本，跑完整评测和灰度。不能只在 prompt 里加一句新规则就直接上线。

## 24. 如何区分模型错误和解释错误？

结论错是模型审核错误，要进入 hard sample；结论对但解释错是 reasoning_error，优先级低一些。业务上先保证处置结论准确，再逐步优化解释质量。

## 25. 这个项目最能体现你能力的点是什么？

不是“我会 SFT”，而是我能把业务风控问题转成可学习的多任务协议，能构造数据闭环，能设计评测和消融，能考虑灰度上线、监控、兜底和人审复核。这是从模型开发到生产落地的完整能力。
