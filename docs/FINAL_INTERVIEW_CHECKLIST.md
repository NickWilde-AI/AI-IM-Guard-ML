# 面试前最终自检清单

这份清单用于面试前 30 分钟快速确认：你不是背项目，而是真的能从 0 到 1 讲清楚。

## 1. 业务理解

- 能否一句话说清项目：IM 私聊审核从文本二分类升级为语义 + 行为多证据审核 Judge。
- 能否解释为什么规则引擎不够：强关键词可以，mid_risk 灰区不行。
- 能否举出灰区例子：文本“今晚老规矩” + 30 分钟大额打赏。
- 能否说清下游消费方：策略系统、人审系统、数据看板、样本回流。

## 2. 输入输出协议

- 能否画出三段输入：`audit_scene / chat_evidence_list / behavior_abnormal_list`。
- 能否说清三层输出：`risk_level / final_judgment / handling_suggestion`。
- 能否解释为什么不能从 risk 直接映射 handling。
- 能否说明 `correlation_analysis` 和 `judgment_basis` 的区别。

## 3. 数据体系

- 能否说出 4 类训练数据：历史工单、合成案例、灰区 hard sample、公开安全数据。
- 能否解释合成数据为什么需要、有什么风险、怎么控制。
- 能否讲清 refinement：上一轮模型误判 -> committee 过滤 -> 回灌。
- 能否说明公开数据只参与二分类，不污染处置标签。
- 能否回答怎么防数据泄漏：ID 集合差、payload/近重复检测。

## 4. Rubric 与标注

- 能否说出 11 类主题中的至少 6 类。
- 能否用代刷/诈骗/私下交易举 low/mid/high 例子。
- 能否解释为什么 risk_level 是有序变量。
- 能否讲清三人标注、多数票、仲裁、一致性指标。

## 5. 模型训练

- 能否说明为什么不用纯 prompt API。
- 能否说明为什么用 Qwen3.5-27B，而不是 8B 或纯 API。
- 能否解释 completion-only loss。
- 能否说明为什么单模型多任务优于三个模型。
- 能否讲清训练超参：2 epoch、2e-6、8192、batch 64。

## 6. 实验结果

- 能否记住主指标：Acc 82.1、risk macro-F1 75.6、handling macro-F1 73.2、ban FPR 2.6%。
- 能否解释为什么 macro-F1 比 micro-F1 更适合。
- 能否说明为什么 ban FPR 是业务红线。
- 能否讲最关键消融：去掉行为证据掉点最大。
- 能否说清 P0/P1 回流测试集为什么重要。

## 7. 生产上线

- 能否画出线上链路：召回证据 -> 行为特征 -> 审核服务 -> vLLM -> JSON 校验 -> 策略路由 -> 人审/处置。
- 能否说明 ban 不直接执行，必须人审复核。
- 能否说出 shadow -> 10% -> 50% -> full 灰度节奏。
- 能否说出回滚条件：ban FPR、客诉率、解析失败率、延迟、特征异常。
- 能否解释版本追踪：model/prompt/rubric/feature schema/postprocess。

## 8. 线上监控

- 能否说出质量指标：人审改判率、申诉成功率、ban FPR。
- 能否说出服务指标：QPS、P95/P99、GPU 显存、queue time。
- 能否说出数据漂移指标：主题分布、risk 分布、礼物金额、行为字段缺失率。
- 能否解释上游行为特征污染会怎样影响模型。

## 9. 高频追问

面试前至少能流畅回答：

- 为什么不是普通文本审核？
- 为什么行为证据是最大贡献？
- 合成数据会不会带偏？
- 怎么防止测试集泄漏？
- JSON 输出坏了怎么办？
- 线上效果和离线不一致怎么排查？
- 如果新增违规主题怎么迭代？
- 这个项目最体现你能力的点是什么？

## 10. 最后一段收束话术

“这个项目不是简单训练一个分类器，而是把 IM 私聊风控做成了完整的业务审核系统。我从业务 taxonomy 和输出协议开始，构造历史工单、合成数据、灰区 hard sample 和公开安全数据，用 Qwen3.5-27B 做 completion-only 多任务 SFT，再通过后处理、策略路由、人审复核、版本追踪和监控回流保证能上线。核心价值是解决 mid_risk 灰区，尤其是语义很淡但行为异常的样本。”
