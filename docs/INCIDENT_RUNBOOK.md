# 线上事故复盘 Runbook

这份文档用于回答“线上出问题你怎么处理”。内容审核模型最怕的不是离线分数低，而是上线后误杀、漏放、延迟、上游特征污染和处置链路异常。

## 1. 总体处理原则

先止血，再定位，再回滚或修复，最后复盘和回流样本。

1. 降低处置强度：暂停 `ban_account` 自动流转，全部进人审。
2. 切换兜底：流量切回上一模型或规则引擎。
3. 固化现场：保存请求、模型输出、版本、特征、路由和人审反馈。
4. 定位原因：按输入、模型、后处理、策略、上游服务逐层排查。
5. 修复验证：离线回放事故样本，再 shadow 灰度。
6. 回流样本：把确认的误判样本进入 refinement 池。

## 2. 事故一：ban FPR 突然升高

现象：

- `ban_account_rate` 高于历史基线。
- 人审驳回率升高。
- 客诉或申诉成功率升高。

止血：

- ban 全部改为 `human_review_required`。
- 小流量切回 shadow。
- 保留 warning 和部分 limit，避免审核链路完全失效。

排查：

- 看 `gift_total_value`、`reward_behavior` 是否异常偏大。
- 看 `behavior_abnormal_list` 是否突然变多。
- 看 `model_version / rubric_version / postprocess_version` 是否变更。
- 抽样看 raw output 是否出现格式异常或过度判高。
- 对比规则引擎和上一版本模型输出。

常见根因：

- 上游行为服务默认值写错。
- 新 rubric 过严。
- 合成数据高风险样本占比过高。
- 后处理把 `limit_account` 错路由成 ban。

修复：

- 修上游特征或回滚特征版本。
- 降低 ban 路由条件。
- 回放事故样本，补充“高风险但证据链不完整不 ban”的 hard sample。

## 3. 事故二：漏放高风险违规

现象：

- P0/P1 工单复盘中模型判安全。
- 规则命中但模型未命中。
- 用户举报或人审发现漏放。

止血：

- 对强规则主题恢复规则优先。
- 对相关 topic 提高召回阈值。
- 将可疑样本进入人审队列。

排查：

- 聊天证据是否被截断。
- `risk_point` 是否缺失。
- 行为异常是否未聚合。
- prompt 是否选错 topic rubric。
- 模型是否对暗语或新话术不敏感。

修复：

- 补短证据列表训练样本。
- 更新 rubric 暗语词和行为模式。
- 加入 hard sample refinement。
- 新增上游召回规则或行为异常类型。

## 4. 事故三：JSON 解析失败率升高

现象：

- `parse_non_ok_rate` 升高。
- 兜底比例上升。
- raw output 多解释文本、缺字段或格式损坏。

止血：

- 启用正则兜底。
- 解析失败默认 `ignore` 或进入规则兜底。
- 暂停依赖模型直接处置的链路。

排查：

- 是否修改 prompt 或 JSON Schema。
- 是否打开 sampling。
- `max_new_tokens` 是否过小导致截断。
- 模型 checkpoint 是否切错。

修复：

- 恢复贪心解码。
- 固定字段顺序。
- 增加格式坏例到 SFT。
- 必要时启用 grammar constrained decoding，但要评估延迟。

## 5. 事故四：P95/P99 延迟超预算

现象：

- vLLM queue time 升高。
- P95 超过 1.2s。
- GPU 显存或 KV cache 压力升高。

止血：

- 限流或降级部分低风险流量到规则。
- 缩短 max_new_tokens。
- 临时扩容实例。

排查：

- 输入 token 是否变长。
- prefix cache 是否失效。
- QPS 是否超过容量规划。
- 是否有异常长证据列表。
- tensor parallel 或 batch 参数是否变更。

修复：

- 恢复 prefix caching。
- 控制证据截断策略。
- 增加实例。
- 对强规则样本做分层路由，减少大模型调用。

## 6. 事故五：行为字段大量缺失

现象：

- `empty_behavior_rate` 升高。
- 模型更多依赖文本，mid_risk 准确率下降。
- limit/ban 占比异常波动。

止血：

- 缺行为字段样本不允许直接 ban。
- 进入规则兜底或人审。

排查：

- 行为特征服务是否超时。
- 上游数据表是否延迟。
- 字段 schema 是否变更。
- 默认值是否被清空或写成字符串。

修复：

- 恢复行为服务。
- 增加字段 schema 兼容。
- 缺行为字段时使用保守 prompt 和保守路由。

## 7. 复盘模板

每次事故复盘至少记录：

- 事故时间线。
- 影响范围：请求量、用户数、处置数。
- 触发指标：ban FPR、解析失败率、延迟、客诉等。
- 涉及版本：model/prompt/rubric/feature schema/postprocess。
- 根因：输入、模型、后处理、策略、上游服务中的哪一层。
- 止血动作。
- 长期修复。
- 回流样本数量和训练计划。

## 8. 面试表达

可以这样说：

“我会把线上问题分成输入异常、模型异常、解析异常、策略异常和服务异常。先止血，比如 ban 全部进人审或切回规则；再用版本追踪和审计日志定位是模型、prompt、rubric、特征还是后处理变了；最后把确认的误判样本回流到 refinement。这个闭环能保证模型不是一次性上线，而是可运营的系统。”
