# 数据接入规范

本文档定义 IM 私聊风控审核系统如何对接上游数据源，包括接入协议、字段规范、质量约束和运维要求。

## 1. 数据源总览

```
┌─────────────┐     ┌──────────────┐     ┌──────────────────┐
│  IM 后端    │────▶│  证据聚合    │────▶│  审核服务         │
│  (gRPC)     │     │  (Kafka/HTTP)│     │  (POST /judge)   │
└─────────────┘     └──────────────┘     └──────────────────┘
       │                    │
       ▼                    ▼
  聊天消息流           行为特征服务
  (chat_evidence)      (behavior_abnormal)
```

## 2. 接入协议

### 2.1 实时接入（推荐）

| 参数 | 值 |
| --- | --- |
| 协议 | HTTP POST (JSON) / gRPC |
| 端点 | `/judge` |
| 超时 | 客户端 3s，服务端 5s |
| 重试 | 最多 1 次，间隔 500ms |
| 幂等 | 基于 ticket_id 去重 |

### 2.2 批量接入（离线评测）

| 参数 | 值 |
| --- | --- |
| 格式 | JSONL 文件 |
| 命令 | `im-guard predict data/input.jsonl` |
| 场景 | 离线评测、回溯审核、数据回流 |

## 3. 输入字段规范

### 3.1 必填字段

| 字段 | 类型 | 说明 | 示例 |
| --- | --- | --- | --- |
| `ticket_id` | string | 审核工单唯一 ID | `"im-20260607-0001"` |
| `chat_evidence_list` | list[string] | 聊天内容证据（已脱敏） | `["加微信稳赚"]` |

### 3.2 可选字段

| 字段 | 类型 | 说明 | 缺失时行为 |
| --- | --- | --- | --- |
| `behavior_abnormal_list` | list[string] | 行为异常证据 | 视为无行为异常 |
| `audit_scene` | string | 场景标识 | 默认 `"private_chat"` |
| `user_intimacy` | string | 用户亲密度 | 默认 `"stranger"` |
| `chat_type` | string | 对话类型 | 默认 `"private"` |
| `user_profile` | dict | 用户画像 | 不参与模型推理 |

### 3.3 上游行为特征服务对接

行为特征由独立的用户行为分析服务产出，通过以下方式注入：

```json
{
  "behavior_abnormal_list": [
    "短时间高频私聊（30 分钟内联系 15 人）",
    "注册不足 24 小时",
    "送礼金额累计超过 500 元"
  ]
}
```

行为特征服务应提供：
- 高频联系检测（阈值：30min 内 > 10 人）
- 新用户标记（注册 < 7 天）
- 异常送礼检测（单日 > 200 元 或 累计 > 500 元）
- 被举报次数（7 天内 > 3 次）
- 设备/IP 异常（多账号同设备）

## 4. 数据质量约束

### 4.1 上游 SLA

| 指标 | 要求 | 降级策略 |
| --- | --- | --- |
| 可用性 | > 99.9% | 降级为纯文本审核（忽略行为特征） |
| 延迟 | P95 < 50ms（证据聚合） | 超时则不等待行为特征 |
| 数据完整性 | ticket_id 和 chat_evidence 必填 | 缺失则拒绝并返回 422 |

### 4.2 输入校验规则

```python
# 在 /judge 端点入口执行
assert ticket_id is not None and len(ticket_id) > 0
assert chat_evidence_list is not None and len(chat_evidence_list) > 0
assert all(isinstance(e, str) for e in chat_evidence_list)
assert total_payload_bytes <= MAX_REQUEST_BYTES (256KB)
```

### 4.3 数据脱敏要求

上游在发送前必须完成以下脱敏：
- 手机号 → `1**********`
- 身份证号 → `***`
- 银行卡号 → `****`
- 真实姓名 → 保留姓氏 + `**`

审核服务不应接收未脱敏的 PII。若检测到 PII 模式（见 `privacy.py`），将在审计日志中标记但不阻断请求。

## 5. 批量数据处理

### 5.1 训练数据收集

```
每日定时任务（T+1）
  → 从审计日志导出前一天已闭单的工单
  → 人工复核过的标记为 internal 来源
  → 自动关闭的标记为 auto_labeled 来源
  → 写入 data/train/daily/ 目录
```

### 5.2 数据回流条件

| 来源 | 入选条件 | 标签来源 |
| --- | --- | --- |
| 人工复核 | reviewer 确认的最终判定 | 人工标注 |
| 用户申诉 | 申诉成功（误判确认） | 修正标签 |
| hard case | committee 分歧 > 阈值 | 专家标注 |
| 漏放举报 | 举报后确认违规 | 补充标注 |

## 6. 监控与告警

### 6.1 接入层监控

| 指标 | 含义 | 告警阈值 |
| --- | --- | --- |
| 请求量 QPS | 系统负载 | 突增 3x 触发扩容 |
| 空 evidence 比例 | 上游异常 | > 5% 告警 |
| 超大 payload 比例 | 异常请求 | > 1% 告警 |
| ticket_id 重复率 | 重复提交 | > 2% 告警 |

### 6.2 数据质量日报

每日自动生成数据质量报告（`im-guard audit-data`），检查：
- 字段缺失率
- 标签分布偏移
- 重复样本比例
- PII 泄露风险

## 7. 部署模式与接入差异

| 模式 | 接入方式 | 适用场景 |
| --- | --- | --- |
| 本地 demo | CLI 直接读取 JSONL | 开发调试 |
| 单机服务 | HTTP POST /judge | 验证和压测 |
| 生产部署 | 网关 → 负载均衡 → 审核服务集群 | 线上流量 |

生产部署时，建议在网关层完成：
- 请求鉴权（API Gateway token 校验）
- 全局限流（基于租户/业务线）
- 请求路由（按场景分发到不同审核服务）
- 请求日志（access log 独立于审计日志）
