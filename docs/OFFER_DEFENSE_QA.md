# Offer 高压答辩 Q&A

这份文档准备的是高薪岗位面试里更尖锐的问题。回答目标不是显得项目“无所不能”，而是显得你成熟、真实、知道边界、能落地。

## 1. 你这个项目是不是只是包装出来的 demo？

不是简单 demo，但也不是我会夸成已经真实上线的系统。  

准确定位是：生产化展示工程。它用公开数据和启发式 Judge 保证仓库可复现，同时保留真实业务接入所需的训练入口、API、安全、审计、监控、部署、人审和灰度治理模块。

我会这样区分：

- 当前仓库可复现：数据转换、审计、预测、评测、API、审计落盘、监控告警、部署模板、readiness-check。
- 真实生产需要替换：内部 IM 私聊数据、真实 SFT checkpoint、人审平台、生产网关、集中审计库和模型注册平台。

这个项目的价值在于证明我知道一个 LLM 风控系统从模型到上线要补哪些工程环节，而不是只会写 prompt。

## 2. 没有真实业务数据，为什么还值得看？

真实业务数据不能公开，这在风控、安全、推荐、广告项目里都很常见。关键是看候选人是否知道数据应该怎么接、怎么审、怎么防污染。

这个项目里我做了三件事：

1. 选了 Apache-2.0 的 XGuard 中文安全数据做公开安全底座。
2. 设计了保守映射：公开违规样本只给 `mid_risk / warning`，不训练 `limit_account / ban_account`。
3. 保留了 internal、synthetic、hard_case 的入口和文档，说明真实上线时如何接历史工单、人审和申诉样本。

所以它不能证明真实线上效果，但能证明数据治理思路是对的。

## 3. 模型指标是不是不能完全复现？

当前 GitHub 仓库不能完全复现真实业务指标，因为它没有内部测试集和真实 checkpoint。  

所以我不会把模型卡指标说成“开源仓库当前跑出来的结果”。我会说：模型卡展示的是企业交付格式、指标体系和目标红线；当前仓库可复现的是工程链路和评测工具。

如果面试官要求现场验证，我会跑：

```bash
make enterprise-check
make predict-route
make eval-report
make window-alerts
make drift-report
```

这证明工程闭环，而不是伪装成真实线上实验。

## 4. 为什么这个项目能支撑 50 万年薪，而不是普通 CRUD？

因为它覆盖的是 LLM 应用工程里最有价值的几个能力：

- 业务抽象：把 IM 风控从文本二分类重构为多证据、多任务审核系统。
- 数据治理：知道公开数据、内部工单、合成样本和 hard case 怎么分工。
- 模型训练：理解 completion-only SFT、多任务输出和强处置标签污染风险。
- 生产工程：API 鉴权、审计、监控、告警、部署、灰度、回滚都考虑到了。
- 风险意识：明确 ban 不直接执行，公开数据不训练强处置，真实上线要人审和平台治理。

50 万岗位通常买的不是“会调库”，而是能把模型做成业务系统的人。

## 5. 项目是不是功能堆太多？

如果只是堆功能，会有这个风险。所以我现在的讲法不是逐个展示模块，而是只围绕一条主线：

> 输入证据 -> LLM Judge -> 结构化输出 -> 策略路由 -> 审计监控 -> 人审回流。

所有模块都服务这条主线：

- 数据审计服务训练可靠性。
- API 鉴权和 request_id 服务线上接入。
- 审计落盘服务追责和复盘。
- Prometheus/window/drift 服务监控和回滚。
- Docker/K8s 服务部署评审。

面试时不主动讲所有文件，只在被追问时展开。

## 6. 如果面试官说“这不就是 prompt engineering”？

我会回答：prompt 是其中一环，但项目核心不是 prompt。

prompt 只解决模型输入格式，解决不了：

- 公开数据如何避免污染强处置标签。
- 三层输出如何一致校验。
- JSON 解析失败如何兜底。
- ban 如何路由到人审。
- 每次判断如何追踪 model/prompt/rubric/schema/postprocess 版本。
- 线上 ban 比例异常如何报警和回滚。

这些才是从 prompt demo 到生产系统的差别。

## 7. 如果面试官说“为什么不用审核 API，自己训练干嘛”？

通用审核 API 可以做底座，但业务处置边界通常不一致。IM 私聊风控不只是判断有害内容，还要判断风险等级和处理动作。

例如同样是“加微信”，普通社交可能是安全，结合收益承诺和短时间批量私聊就可能是诈骗引流。业务模型需要学习平台自己的证据组合和处置边界。

合理方案不是完全不用审核 API，而是：

- 公开安全模型/API 做基线和辅助标注。
- 业务 SFT 学习 IM 场景的三层输出。
- 规则和人审做强处置兜底。

## 8. 你怎么证明行为证据有用？

最直接是消融实验：去掉 `behavior_abnormal_list` 和 `behavior_key_summary`，比较 risk macro-F1、handling macro-F1 和 ban FPR。

在当前开源仓库里，我能展示行为证据如何进入输入协议、prompt、启发式 Judge、监控和文档；真实业务里要用固定 eval set 做消融。

面试回答要强调：不是说行为证据天然有用，而是要通过消融证明，并且要监控上游特征污染。

## 9. 线上出问题你怎么排查？

按链路排查，不直接甩锅模型：

1. 输入分布：主题、risk、行为字段缺失率是否漂移。
2. 证据构造：聊天证据是否被截断，行为摘要是否默认值异常。
3. 模型输出：JSON 解析失败率、fallback 率是否升高。
4. 版本一致性：model/prompt/rubric/schema/postprocess 是否变更。
5. 策略路由：ban 是否被错误自动执行，是否绕过人审。
6. 审计复盘：按 request_id 和 ticket_id 查输入摘要、版本和结果。

项目里对应有 `/metrics`、`window-alerts`、`drift-report`、SQLite/JSONL 审计和版本字段。

## 10. 这个项目最薄弱的地方是什么？

最薄弱的是没有真实业务数据和真实 checkpoint。这个问题不能回避。

但我做了两个补救：

1. 用公开数据跑通可复现的数据接入和标签映射。
2. 把真实生产所需的接口、文档和治理流程补齐，说明只要接入内部数据和 checkpoint 就能进入生产评审。

如果进入公司，我第一优先级会是建立固定 eval set 和人审回流闭环，而不是继续加功能。

## 11. 如果只给你两周继续做，你做什么？

我不会继续堆模块，会做三件最有业务价值的事：

1. 建立 500-1000 条人工复核 eval set，覆盖 11 类主题和 high-impact 处置。
2. 用真实或半真实业务样本训练一个小 LoRA baseline，跑出可复现指标。
3. 接一版最小人审回流：模型建议 -> 人审改判 -> hard case 入池 -> 下轮评测。

这三件事比再加一个 dashboard 或新部署模板更值钱。

## 12. 面试最后怎么收束？

> 这个项目我最想证明的是：我不是只会调模型，而是能把 LLM 放进真实风控链路。  
> 从业务 taxonomy、标签体系、数据审计，到 SFT、结构化输出、策略路由、人审复核、审计、监控和灰度回滚，我都按生产系统的思路做了拆解。  
> 当前仓库是可复现的生产化展示版本；真实上线需要接入内部数据、checkpoint 和企业平台。这个边界我很清楚。
