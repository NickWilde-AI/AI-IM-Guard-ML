# 面试口语实战稿

这份稿子用于真实面试开口说。目标不是逐字背，而是把结构练到稳定：先讲业务，再讲方案，再讲工程边界。

## 1. 20 秒自我引入

> 我这个项目做的是 IM 私聊风控审核。它不是普通文本分类，而是把聊天语义、行为异常、审核场景合在一起，让模型输出风险等级、违规判定、处置建议和解释依据。  
> 我重点做了两块：一块是数据和训练体系，另一块是服务化落地，包括 API、审计、监控、告警、灰度和部署。

如果面试官时间很紧，直接接一句：

> 简单说，它证明我能把 LLM 从一个模型调用，做成一个可评测、可审计、可回滚的业务审核系统。

## 2. 60 秒版本

> 背景是直播/社交场景里的 IM 私聊审核。原来很多审核系统只看文本，做 safe/unsafe 二分类，但真实风控里很多风险不是靠一句话能判断的，比如诈骗引流、代刷包榜、私下交易，经常要结合短时间高频私聊、大额打赏、异常进房这些行为证据。
>
> 所以我重新定义了任务：输入是 `audit_scene`、`chat_evidence_list`、`behavior_abnormal_list` 三段证据；输出不是简单二分类，而是 `risk_level`、`final_judgment`、`handling_suggestion`，再加关联分析和判定依据。这样下游策略、人审和监控都能消费。
>
> 工程上，我接入了公开 XGuard 数据，做了保守标签映射，避免公开二分类数据污染 ban 这种强处置；然后补了数据审计、训练入口、离线评测、API 鉴权、request_id、JSONL/SQLite 审计、Prometheus 指标、滑动窗口异常、drift 检测、Docker/K8s 和 readiness-check。
>
> 当前仓库是生产化展示版，不包含公司内部私聊数据和真实 checkpoint，但它完整保留了真实业务接入的工程骨架。

## 3. 2 分钟版本

> 这个项目我想解决的是 IM 私聊审核里三个问题：第一，只看文本不够；第二，只判 safe/unsafe 不够；第三，模型输出如果不能接策略、人审和审计，那就不能上线。
>
> 第一层是业务建模。我把输入拆成审核场景、聊天证据和行为异常。比如一句“加微信稳赚”本身是语义证据，短时间高频私聊就是行为证据，两者结合才更像诈骗引流。输出拆成三层：风险等级、是否违规、处置建议。这样 high risk 不一定直接 ban，ban 也必须经过人审或策略保护。
>
> 第二层是数据治理。我选了 Apache-2.0 的 XGuard 中文安全数据做公开安全底座，但它不是 IM 私聊真实数据，所以我做了保守映射：公开违规样本只转成 `mid_risk / warning`，不产生 `limit_account` 和 `ban_account`。同时加了字段缺失、标签非法、重复样本、训练评测泄漏和 PII 风险检查。
>
> 第三层是工程落地。API 支持 token 和最小 RBAC，响应和审计都有 request_id；审计可以写 JSONL 或 SQLite，并且只保存脱敏输入摘要；监控有 Prometheus 指标、窗口异常检测和 drift 报告；部署上有 Docker、Compose、K8s 和生产环境 env 示例。
>
> 所以这个项目不是想证明我已经有真实线上数据，而是证明我知道一个 LLM 风控审核系统从数据、训练、评测到上线治理应该怎么搭。

## 4. 5 分钟版本结构

按这个顺序讲，不要乱：

1. 业务痛点：文本二分类不够，IM 风控需要行为证据和差异化处置。
2. 任务定义：三段输入、三层输出、解释字段。
3. 数据体系：internal、public_binary、synthetic、hard_case；公开数据不训练强处置。
4. 模型训练：completion-only SFT，多任务输出，LoRA 入口。
5. 评测体系：binary、macro-F1、ban FPR、解析失败率、drift。
6. API 工程：鉴权、request_id、限流、结构化错误、审计查询。
7. 监控部署：Prometheus、window-alerts、drift-report、Docker/K8s。
8. 真实边界：当前是生产化展示，真实上线需要内部数据和企业平台。

收束句：

> 我最想体现的是从业务问题到生产系统的完整闭环，而不是单点模型能力。

## 5. 卡壳时的救场句

如果忘了细节：

> 我先从链路上回答：输入、模型输出、后处理、审计监控、回流，这几个点是连在一起的。

如果被问到真实数据：

> 这个问题要分开说：当前 GitHub 仓库不能公开内部数据；但我在仓库里保留了公开数据接入、保守映射和内部数据接入入口，真实上线时替换数据源即可。

如果被问到模型指标：

> 当前仓库可复现的是工程链路和评测工具；模型卡里的指标是企业交付格式和目标红线，不应该被说成开源仓库现场可复现的线上结果。

如果被问到是不是堆功能：

> 我不按模块讲，我按生产链路讲。每个模块都对应一个上线问题：鉴权解决接入安全，审计解决追责，监控解决异常发现，灰度回滚解决误杀风险。

## 6. 对不同面试官的讲法

### 算法面

重点讲：

- 为什么三层标签。
- 为什么 completion-only。
- 为什么多任务优于三个模型。
- 怎么做消融和 ban FPR。
- 公开数据为什么只做二分类辅助。

少讲：

- Docker/K8s 细节。
- 太多 API endpoint。

### 工程面

重点讲：

- FastAPI 服务边界。
- request_id 和审计链路。
- 结构化错误、限流、鉴权。
- Prometheus、window-alerts、drift-report。
- Docker/Compose/K8s 和 readiness-check。

少讲：

- 大模型参数细节。
- 指标故事过长。

### 业务/主管面

重点讲：

- 为什么二分类不够。
- 为什么 ban 必须人审。
- 如何降低误杀和客诉。
- 如何灰度、回滚、复盘。
- 项目怎么体现端到端 owner 能力。

少讲：

- 训练超参。
- 代码模块名。

## 7. 不能主动说的内容

不要主动说：

- “已经真实生产上线。”
- “模型指标完全可复现。”
- “公开数据等价真实业务数据。”
- “SQLite/Bearer token 就是生产级方案。”

可以主动说：

- “这是生产化展示工程。”
- “真实上线要接内部数据、checkpoint、人审平台和网关。”
- “当前仓库证明工程链路和治理意识。”

## 8. 结束前反问面试官

如果面试官问“你有什么问题”，可以问：

1. 你们现在的内容安全/风控系统更偏规则、传统模型，还是 LLM Judge？
2. 强处置比如封号，是模型直接执行，还是有人审/策略复核？
3. 线上更关注召回、误杀、延迟，还是审计合规？
4. 如果我加入，最希望我先解决数据闭环、模型效果，还是服务稳定性？

这些问题能把你从“求职者”拉到“未来同事一起解决问题”的状态。
